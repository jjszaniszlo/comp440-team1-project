- Each blog has 5-10 root-level comments
- Comments support RECURSIVE NESTING up to 5 levels deep (but most commonly 1-2 levels)
- Each comment (at any level) must have:
  - author_username: Valid username from users array
  - content: 10-250 characters of relevant comment text
  - sentiment: Either "positive" or "negative" EXCLUSIVELY "positive" or "negative"
  - replies: Array of 0-3 nested reply objects (each reply can also have its own replies)

- **IMPORTANT NESTING RULES:**
  - Root comments (level 0): author_username CANNOT be the blog author

  - Most comment threads should be 1-2 levels deep (70-80% of comments)
  - Some threads can go 3-4 levels deep (15-25% of comments)
  - Very few threads should reach the maximum 5 levels (5-10% of comments)

## Why FastAPI Outperforms Flask for Modern APIs

I've been building REST APIs with Python for 5 years, and FastAPI has completely changed my workflow. After migrating three production services from Flask to FastAPI, I've seen response times drop by 40% and deployment bugs decrease by half. Here's what makes it special.

### Automatic Data Validation

FastAPI uses Pydantic models for request/response validation. This means type hints aren't just documentation - they're enforced at runtime:

\`\`\`python
from pydantic import BaseModel
from typing import Optional

class User(BaseModel):
    username: str
    email: str
    age: int
    bio: Optional[str] = None

@app.post("/users")
async def create_user(user: User):
    # Validation already happened - age is guaranteed to be an int
    if user.age < 13:
        raise HTTPException(status_code=400, detail="Must be 13+")
    return {"message": f"Created user {user.username}"}
\`\`\`

If someone sends `{"username": "john", "age": "twenty"}`, FastAPI automatically returns a 422 with:

\`\`\`json
{
  "detail": [{
    "loc": ["body", "age"],
    "msg": "value is not a valid integer",
    "type": "type_error.integer"
  }]
}
\`\`\`

No more manual validation code. No more `if not isinstance(age, int)` checks scattered everywhere.

### Async Support is First-Class

Unlike Flask, FastAPI is built on Starlette, giving you async/await support out of the box. This matters when you're making external API calls:

\`\`\`python
import httpx

@app.get("/user-with-posts/{user_id}")
async def get_user_with_posts(user_id: int):
    async with httpx.AsyncClient() as client:
        # These run concurrently
        user_response, posts_response = await asyncio.gather(
            client.get(f"https://api.example.com/users/{user_id}"),
            client.get(f"https://api.example.com/posts?user={user_id}")
        )
    return {
        "user": user_response.json(),
        "posts": posts_response.json()
    }
\`\`\`

With Flask, these calls would block. With FastAPI, your server can handle other requests while waiting for I/O. On our production service, this change alone increased throughput from 100 req/s to 280 req/s.

### Automatic API Documentation

Every endpoint you write automatically appears in interactive docs at `/docs`. Swagger UI is built in. No configuration needed:

- Request/response schemas auto-generated from Pydantic models
- Try-it-out functionality works immediately
- OAuth2 flows supported out of the box

Our frontend team loves this because they don't need to ask "what fields does this endpoint accept?" - they just check `/docs`.

### Dependency Injection That Makes Sense

FastAPI's dependency injection is elegant. Want to require authentication?

\`\`\`python
from fastapi import Depends

async def get_current_user(token: str = Depends(oauth2_scheme)):
    # decode token, get user from DB
    return user

@app.get("/me")
async def read_current_user(user: User = Depends(get_current_user)):
    return user
\`\`\`

Dependencies compose. You can depend on other dependencies. Testing becomes trivial because you can override any dependency with a mock.

### Performance Benchmarks

I ran wrk benchmarks against identical endpoints in Flask and FastAPI:

| Framework | Req/s | Latency p50 | Latency p99 |
|-----------|-------|-------------|-------------|
| Flask     | 1,240 | 78ms        | 340ms       |
| FastAPI   | 2,890 | 32ms        | 95ms        |

These numbers are from a real endpoint that queries PostgreSQL and makes one external API call. FastAPI's async nature shines when I/O is involved.

### The Migration Cost

Migrating isn't free. We spent about 2 weeks per service:

- Rewriting route decorators (straightforward)
- Converting to async/await (most of the work)
- Updating tests (FastAPI's TestClient is great though)
- Retraining the team on async patterns

Was it worth it? Absolutely. The performance gains paid for themselves in reduced infrastructure costs within 3 months.

### When Flask Still Makes Sense

FastAPI isn't always the answer. Stick with Flask if:

- Your app is synchronous and always will be
- You're building a traditional server-rendered web app (FastAPI is API-focused)
- Your team isn't comfortable with async/await yet

But for modern API development, especially microservices, FastAPI is the clear winner.
